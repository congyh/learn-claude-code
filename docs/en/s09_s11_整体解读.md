## 核心架构

整个系统由三层构成：**主循环（Lead）**、**TeammateManager**、**MessageBus**，通过文件系统协调多个 LLM 线程。

---

## 1. 基础层（s09）：线程 + 文件信箱

### 进程模型

```
主进程（Lead）
  ├── Thread: alice._teammate_loop()   ← 独立 LLM agent
  ├── Thread: bob._teammate_loop()     ← 独立 LLM agent
  └── .team/
        ├── config.json               ← 花名册 + 状态
        └── inbox/
              ├── alice.jsonl         ← append-only 信箱
              ├── bob.jsonl
              └── lead.jsonl
```

每个 teammate 是一个**独立线程**，持有自己的 `messages[]` 历史，跑自己的 `client.messages.create()` 循环。

### MessageBus：追加写入 + 读后清空

```python
# 发送：追加一行 JSON
open("alice.jsonl", "a").write(json.dumps(msg) + "\n")

# 读取：一次性读走并清空（drain）
msgs = [json.loads(l) for l in path.read_text().splitlines()]
path.write_text("")   # 清空，防止重复消费
```

这是整个通信的核心：**文件就是队列**，无需消息中间件。

### Teammate 的每轮 LLM 调用前，先读信箱

```python
for _ in range(50):
    inbox = BUS.read_inbox(name)
    for msg in inbox:
        messages.append({"role": "user", "content": json.dumps(msg)})
    response = client.messages.create(...)   # 注入了新消息后再推理
```

消息被当作普通 user turn 注入上下文，LLM 自然感知到通信内容。

---

## 2. 协议层（s10）：request_id 关联的握手

s10 引入了 **同一 FSM 驱动两种协议** 的模式：

```
pending → approved
       → rejected
```

两种协议（关闭握手 / 计划审批）结构完全一样：

| 步骤 | 关闭协议 | 计划审批协议 |
|------|----------|--------------|
| 发起方 | Lead | Teammate |
| 请求内容 | `shutdown_request` + `request_id` | `plan_approval` + `request_id` |
| 响应内容 | `shutdown_response` + `approve: bool` | `plan_approval_response` + `approve: bool` |
| 状态追踪 | `shutdown_requests[req_id]` | `plan_requests[req_id]` |

`request_id` 是一个 8 位 UUID 片段，保证跨线程的消息对应关系。

---

## 3. 自主层（s11）：IDLE 轮询 + 任务抢占

s11 的关键改进是 **两阶段生命周期**：

```
WORK 阶段（跑 LLM 循环）
  │
  │ stop_reason != tool_use 或 调用 idle()
  ▼
IDLE 阶段（每 5 秒轮询，最长 60 秒）
  ├─ 检查信箱 → 有新消息 → 回 WORK
  ├─ 扫描 .tasks/ → 有未认领任务 → claim → 回 WORK
  └─ 超时 60s → SHUTDOWN
```

任务抢占用文件锁保证原子性：

```python
with _claim_lock:                    # threading.Lock
    if task.get("owner"): return "Error: already claimed"
    task["owner"] = owner
    task["status"] = "in_progress"
    path.write_text(json.dumps(task))  # 原子写
```

**身份重注入**（解决上下文压缩后遗忘自身身份的问题）：

```python
if len(messages) <= 3:   # 消息太短说明被压缩了
    messages.insert(0, {"role": "user", "content": f"<identity>You are '{name}'...</identity>"})
    messages.insert(1, {"role": "assistant", "content": f"I am {name}. Continuing."})
```

## 整体数据流

```
用户输入
    │
    ▼
Lead agent_loop()
    │  spawn_teammate("alice", "coder", "fix auth bug")
    ▼
TeammateManager.spawn()
    ├── 写 config.json（status: working）
    └── 启动 Thread → alice._teammate_loop()
              │
              │ 每轮 LLM 调用前：read_inbox() → 注入 messages
              │ LLM 调用 send_message() → BUS.send() → bob.jsonl
              │ LLM 调用 idle() → 进入 IDLE 轮询
              │ 60s 无新工作 → status: shutdown → 线程退出
              ▼
         .team/config.json（持久化状态）
         .team/inbox/*.jsonl（临时通信队列）
         .tasks/task_N.json（任务看板）
```

**核心设计哲学**：用文件系统模拟消息队列，每个 LLM 实例是独立线程，通过"在 LLM 调用前注入信箱内容"实现异步通信，整套系统无需任何专门的消息中间件。

## 问题: 这套系统有哪些缺陷?

基于源码，这套系统有以下几类明显缺陷：

---

### 1. 并发安全问题（最严重）

**config.json 无写锁**：多个 teammate 线程共享同一个 `TeammateManager` 实例，都会调用 `_save_config()` 写同一个文件，且没有任何 mutex 保护 `self.config` 字典：

```python
def _save_config(self):
    self.config_path.write_text(json.dumps(self.config, indent=2))  # ← 多线程竞争写
```

**信箱 read-drain 非原子**：`read_inbox()` 先读文件、再清空文件，两步之间如果有新消息写入，会被静默丢弃：

```python
msgs = [json.loads(l) for l in path.read_text().splitlines()]
path.write_text("")   # ← 这两步之间写入的消息永久丢失
```

**`_claim_lock` 是进程内锁**：`threading.Lock()` 只能防同进程内的竞争，无法防多进程场景。

---

### 2. 消息可靠性问题

**消息丢失无感知**：drain-on-read 设计意味着消息被读走就消失，若线程在处理途中崩溃，消息永久丢失，没有任何 ack 机制。

**消息可能乱序**：多个线程并发 `open(inbox, "a").write(...)` 在理论上（大消息超过 PIPE_BUF）可能导致两条消息的字节交错，使 JSONL 解析失败。

**Lead 不自动轮询信箱**：主线程的 `agent_loop` 不会主动检查 `lead.jsonl`，只有 LLM 主动调用 `read_inbox` 工具时才能收消息，可能长时间忽视来自 teammate 的回复。

---

### 3. 状态管理脆弱

**进程崩溃后状态失真**：`config.json` 会持久化 `status: "working"`，但 LLM 对话历史（`messages[]`）只在内存中。进程重启后 config.json 显示 teammate 还在 working，但实际线程已不存在——僵尸状态。

**Daemon 线程被强杀**：所有 teammate 线程都是 `daemon=True`，主进程退出时线程被立即终止，可能在文件写入中途被杀，导致文件损坏。

**身份重注入启发式太脆弱**：

```python
if len(messages) <= 3:   # ← 用消息数量猜测是否发生了上下文压缩
    messages.insert(0, make_identity_block(...))
```

这个判断完全依赖消息数量，而非真实的压缩事件，容易误判。

---

### 4. 协议设计问题

**计划审批没有真正的阻塞**：Teammate 提交计划后收到 `"Waiting for approval."` 字符串，但 LLM 不会真正等待——它会继续推理，可能在审批结果到来前就自行行动。

**关闭协议是建议性的**：如果 teammate LLM 拒绝 shutdown，Lead 没有任何强制终止机制，线程会继续运行。

**shutdown_requests / plan_requests 是进程内字典**：这些 tracker 不持久化，进程重启后所有 pending 请求的状态全部丢失。

---

### 5. 扩展性问题

**信箱无背压**：发送方可以无限追加消息，接收方慢时消息无限堆积，没有流量控制。

**IDLE 是忙等轮询**：每 5 秒 `time.sleep(5)` 检查文件，是对事件驱动的粗糙模拟，无法及时响应。

**单机单进程假设**：整个架构（线程、内存 tracker、Lock）强绑定于单一 Python 进程，无法分布式部署。

---

### 总结

| 层级 | 核心缺陷 |
|------|----------|
| 并发 | config.json 无锁、drain 非原子 |
| 可靠性 | 消息丢失无 ack、daemon 线程强杀 |
| 状态 | 崩溃后僵尸状态、历史不持久化 |
| 协议 | 计划审批无真正阻塞、shutdown 无强制 |
| 扩展 | 忙等轮询、单进程绑定 |

这套系统适合学习演示，但要生产化需要至少解决**并发写锁**和**消息持久化/ack**这两个最根本的问题。

## 问题: agent teams这种工作模式, 有哪些好的开源解决方案?

以下是与 s09-s11 这套模式最相关的开源方案，按相似度和成熟度排列：

---

### 直接对标：Multi-Agent 框架

#### 1. AutoGen（Microsoft）
**最接近 s09-s11 的设计**。核心概念几乎一一对应：

| s09-s11 概念 | AutoGen 对应 |
|---|---|
| Lead + Teammates | `UserProxyAgent` + `AssistantAgent` |
| JSONL 信箱 | `GroupChat` 消息历史 |
| `spawn_teammate` | `initiate_chat()` |
| `broadcast` | `GroupChatManager` |

支持 human-in-the-loop、代码执行沙箱、结构化终止条件。

```
pip install pyautogen
```

#### 2. CrewAI
**最接近 s11 任务看板模式**。明确的 Role + Task 抽象：

```python
coder = Agent(role="Senior Coder", goal="...", backstory="...")
task = Task(description="Fix auth bug", agent=coder)
crew = Crew(agents=[coder, tester], tasks=[tasks], process=Process.hierarchical)
crew.kickoff()
```

`Process.hierarchical` 模式有明确的 manager agent 分配任务，`Process.sequential` 则是流水线。

#### 3. OpenAI Swarm
**最接近 s09 的极简实现**，官方定位就是"教学用"。用 `handoff` 函数替代信箱通信，核心只有 200 行，适合直接阅读源码理解原理。

#### 4. LangGraph（LangChain）
**最生产级**。用有向图描述 agent 的状态转换，原生支持：
- 持久化 checkpoint（解决了 s09 崩溃后状态丢失问题）
- human-in-the-loop 中断（解决了 s10 计划审批阻塞问题）
- 分布式执行（LangGraph Platform）

---

### 解决具体缺陷的基础设施

| s09-s11 缺陷 | 对应工具 |
|---|---|
| 消息丢失、drain 非原子 | **Celery + Redis**（持久队列 + ack） |
| config.json 无写锁 | **Redis** 原子操作 / **SQLite WAL** 模式 |
| 单进程绑定 | **Ray**（分布式 actor 模型） |
| 忙等轮询 | **Redis Pub/Sub** / **asyncio.Event** |

---

### 选型建议

```
学习原理         → OpenAI Swarm（最简）或继续读 s09-s11 源码
快速搭原型       → CrewAI（概念清晰，5分钟跑起来）
对话式 Agent     → AutoGen（最接近 s09 信箱模型）
生产级工作流     → LangGraph（状态持久化 + 人工审批原生支持）
高并发分布式     → Ray + 自定义 actor
```

这几个框架本质上都是在解决同一个问题：**如何让多个 LLM 调用之间共享状态、分工协作**。s09-s11 用文件系统手工实现了一遍，读懂之后再看这些框架会非常清晰。
